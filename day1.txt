‚óŹ The project is a fresh repo with no existing files. Before implementing,
  here's the tradeoff you asked about:

  ---
  Three options for frozen-6/mutable-1 in Pydantic v2:

  A. __setattr__ override (my recommendation)
  Override __setattr__ to raise AttributeError for any frozen field name.
  Pydantic v2's generated __init__ uses object.__setattr__ internally, so
  initialization bypasses the guard ‚ÄĒ only post-construction writes hit it.
  - Pro: Hard runtime enforcement, loud immediate failure, single class, zero
  boilerplate
  - Con: Mypy/pyright won't flag order.price = 50 at type-check time ‚ÄĒ only
  catches it at runtime

  B. Code discipline + docstring
  frozen=False, document which fields are off-limits, rely on callers to comply.
  - Pro: Nothing to implement
  - Con: No enforcement whatsoever

  C. Frozen core model + mutable wrapper
  frozen=True Pydantic model for the 6 frozen fields; a separate class holding
  that model plus remaining.
  - Pro: Type-checker enforces immutability of the frozen model's fields
  - Con: Two objects, delegation boilerplate (order.core.price vs order.price
  unless you proxy), more complexity than a toy warrants

  Property with no setter (you mentioned this): doesn't work cleanly in Pydantic
  v2 ‚ÄĒ field descriptors shadow @property overrides on model fields.

  My pick: A. For a matching engine, you want a loud crash the instant a frozen
  field is touched ‚ÄĒ silent mutation of price or sequence_number is a
  correctness bug that would be extremely subtle. Runtime enforcement beats
  type-only enforcement here because the bug manifests immediately in dev/test
  rather than only if someone runs a type checker.

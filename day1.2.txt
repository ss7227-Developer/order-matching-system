‚óŹ Good instinct to ask. Let me walk through the concurrency problem honestly
  before writing anything.

  ---
  The core challenge

  The lock's job is to make races not observable. If the lock works, threads
  serialize, and the test sees clean state. If you remove the lock, you might
  see corruption ‚ÄĒ but Python's GIL complicates things. The GIL ensures
  individual bytecodes are atomic, so a single dict __setitem__ or deque append
  can't tear. What it doesn't protect is multi-step composite operations.

  The real race this lock guards is: Thread A is mid-match, iterating _level()'s
  deque, while Thread B's cancel_order does remove() ‚ÄĒ which modifies that same
  deque AND _index in two separate steps. Without the lock, Thread A could see
  a half-removed order (in the deque but gone from _index, or vice versa).

  What the test can actually prove

  It cannot guarantee it catches a race ‚ÄĒ race windows are narrow and
  non-deterministic. What it can prove:

  1. No exceptions raised ‚ÄĒ no KeyError, ValueError, RuntimeError during
  concurrent access
  2. Structural invariants hold after all threads join ‚ÄĒ every order_id in
  _index is present in exactly one price-level deque at the right side/price,
  and every order in any deque is in _index. If those are ever both true
  simultaneously, the two-step remove didn't half-execute.

  Why it would likely catch a real bug

  If you removed the lock and ran enough iterations, Thread A's _match() loop
  could call self._book.remove(resting) on an order that Thread B already
  removed. remove() does book[order.price].remove(order) (raises ValueError if
  not found) then _index.pop(order_id, None). The ValueError from deque removal
  would propagate and get caught by the thread exception collector.

  What it won't catch

  Subtle reordering bugs that only produce wrong trades (not exceptions). That
  would require deterministic interleaving ‚ÄĒ not achievable without a mock
  scheduler. The test catches corruption, not logical incorrectness under
  concurrency.

  Concrete setup

  - Thread 1: submits 80 sell orders at varying prices, puts their IDs on a
  queue.Queue
  - Thread 2: drains that queue and calls cancel_order on each ID (many will
  already be filled ‚ÄĒ that's fine, cancel_order returns None safely)
  - Thread 3: submits 80 buy orders at crossing prices, causing matches against
  Thread 1's resting orders
  - Each thread collects its exceptions into a shared list
  - After join: assert no exceptions, then walk _book._bids, _book._asks, and
  _book._index to verify cross-consistency
